## 简介 Introduction

### 目标 Aims

- 用户密码绝不能在网络上传播，也绝不能以任何形式存储在客户端机器上：使用后必须立即丢弃
- 即使在身份验证服务器数据库中，用户密码也不应该以未加密的形式存储；
- 每个工作会话仅要求用户输入一次密码。此特性称为单点登录；
- 认证信息管理是集中的，驻留在认证服务器上。应用程序服务器不得包含其用户的身份验证信息
 - 管理员可以通过在单个位置执行操作来禁用任何用户的帐户，而无需在提供各种服务的多个应用程序服务器上执行操作；
 - 当用户更改密码时，所有服务的密码都会同时更改；
 - 没有冗余的认证信息，否则必须在各个地方进行保护；
- 用户不仅必须证明他们是他们所说的人，而且在请求时，应用程序服务器也必须向客户端证明他们的真实性。这种特性被称为相互认证；
- 身份验证和授权完成后，如果需要，客户端和服务器必须能够建立加密连接。为此，Kerberos 支持生成和交换用于加密数据的加密密钥。

### 定义 Definition 

#### 主体 Principal

*`component1/component2/.../componentN@REALM`*

**imap/mbox.example.com@EXAMPLE.COM**

#### 票证 Ticket

由身份验证服务器颁发，并使用它们所针对的服务的密钥进行加密。由于此密钥是仅在身份验证服务器和提供服务的服务器之间共享的秘密，因此即使是请求票证的客户端也无法知道或更改其内容。

#### 加密 Encryption

- Windows Active Directory
RC4-HMAC、DES(56)
- Unix
RC4-HMAC、DES(56)、3DES、AES128、AES256

先使用string2key对原始密钥做散列，保证长度一致，散列算法比如：MD5、CRC32

```
# 使用用户主体作为盐
salt = principal
encrypted = string2key(unencrypted + salt)
```
**密钥版本号（kvno）**：当用户更改密码或管理员更新应用程序服务器的密钥时，此更改会通过增加计数器来记录。标识密钥版本的计数器的当前值称为密钥版本号或更简称为kvno。

#### 密钥分发中心 Key Distribution Center

分为三个部分：数据库、身份验证服务器（Authentication Server AS）和票证授予服务器（Ticket Granting Server TGS）

用户输入密码后，AS发出一个特殊的票证，称为票证授予票证，或更简称为 TGT，与之关联的主体是 krbtgt/REALM@REALM。TGS可以向具有有效 TGT 的客户端分发服务票证，其中包含了可以用于后续会话的会话密钥。
客户端请求服务器时，除了包含票证的请求外，还需添加另一个数据包（身份验证器），其中包含用户主体和当时时间戳，并使用会话密钥对其进行加密；服务器在收到此请求后，解压缩第一张票，提取会话密钥，如果用户确实是所说的人，服务器能够解密提取时间戳的身份验证器。如果后者与服务器时间相差小于 2 分钟（但可以配置容差），则身份验证成功。
客户端永远不会保留用户的密码，也不会记住通过应用 string2key 获得的密钥：它们用于解密来自 KDC 的回复并立即丢弃。但是会记住票证和相关会话密钥

#### 重放缓存 Replay Cache

冒名顶替者有可能同时窃取票和验证器，并在验证器有效的 2 分钟内使用它们。为解决这一可能性很低的问题，引入重放缓存。在应用程序服务器（以及TGS）中，存在记住最近 2 分钟内到达的验证器的能力，如果它们是副本则拒绝它们。


### 操作 Operation

- **AS_REQ**：初始用户身份验证请求(`使用kinit发出`)。包含请求用户的密钥
- **AS_REP**：AS对先前请求的回复。包含 TGT（使用 TGS 密钥加密）和会话密钥（使用请求用户的密钥加密）
- **TGS_REQ**：客户端向TGS请求服务票证。包含从先前消息中获得的 TGT 和客户端生成并使用会话密钥加密的身份验证器
- **TGS_REP**：TGS对先前请求的回复。包含请求的服务票据（用服务的秘钥加密）和TGS生成的服务会话密钥，使用AS之前生成的会话密钥加密；
- **AP_REQ**：客户端向应用服务器发送的访问服务的请求。包含从 TGS 获得的服务票证，和客户端再次生成的身份验证器（使用服务会话密钥加密）
- **AP_REP**：应用服务器给客户端的回复，以证明它确实是客户端所期望的服务器。客户端仅在需要相互验证时才向服务器请求它。
